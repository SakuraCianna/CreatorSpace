# CreatorSpace 需求文档

版本: v0.4 终极形态规划稿
日期: 2026-06-18
状态: 已补充终极产品目标、最终数据库边界与模块分工原则

## 1. 项目定位

CreatorSpace 是一个基于 Spring Boot 与 Vue 3 的个人主题风格博客与创意作品展示平台。它不只承担文章发布功能，还要作为个人品牌展示、创意项目陈列、内容管理和数据化运营的综合内容系统。

平台核心方向是“内容系统”，不是单一博客 CRUD。系统需要同时支持文章、作品集、主题配置、创意墙、评论互动、访问统计、文件资源、搜索和后台 CMS 管理。

### 2.3 官方资料校验

- Spring Boot 当前官方系统要求页面显示 Spring Boot 4.x 需要 Java 17 及以上，当前选择 Java 21 LTS 可满足要求。参考: https://docs.spring.io/spring-boot/system-requirements.html
- Spring Boot 4.0 发布说明提到其基于新一代 Spring 生态、支持 Java 25，同时保留 Java 17 兼容性。参考: https://spring.io/blog/2025/11/20/spring-boot-4-0-0-available-now
- Vite 官方文档要求 Node.js `20.19+` 或 `22.12+`，当前 Node 24 满足要求。参考: https://vite.dev/guide/
- Vue 官方快速开始同样要求 Node.js `^20.19.0 || >=22.12.0`。参考: https://vuejs.org/guide/quick-start
- MyBatis-Spring 4.0 对应 Spring Framework 7.0+ 和 Java 17+，与 Spring Boot 4.x 的 Spring 7 基线匹配。参考: https://mybatis.org/spring/
- MyBatis-Plus 官方安装文档提示 3.5.9+ 开始插件部分改为可选依赖，后续接入分页插件时需要显式选择对应插件支持模块。参考: https://baomidou.com/en/getting-started/install/
- Material Design 3 是 Google 当前开放设计系统，后台 CMS 视觉与交互规范按该体系设计。参考: https://m3.material.io/
- PostgreSQL `pg_trgm` 官方文档说明其支持相似度匹配和可用于快速搜索的索引操作类，适合作为第一阶段站内搜索优化方案。参考: https://www.postgresql.org/docs/current/pgtrgm.html
- 智谱官方文档显示 `glm-4.6v-flash` 是免费多模态模型，支持视频、图像、文本和文件输入，接口使用 `https://open.bigmodel.cn/api/paas/v4/chat/completions`。参考: https://docs.bigmodel.cn/cn/guide/models/free/glm-4.6v-flash

### 2.4 已确认技术决策

1. 后端数据访问层使用 MyBatis-Plus，并接入 MyBatis-Plus 分页插件。
2. 后台 UI 采用 Google Material Design 3 设计语言，优先按 Material 组件、色彩、密度、状态和动效规范设计。
3. 文件存储第一阶段使用本机本地存储，存储根目录使用相对项目根目录的 `storage/uploads`，不接入 MinIO / S3。
4. 鉴权方案使用 JWT，本地 `.env` 已生成随机 JWT 密钥。
5. 搜索方案选择 PostgreSQL `pg_trgm` + GIN 索引，优先保证第一阶段搜索性能。
6. 第一阶段需要注册功能，同时内置管理员账号 `admin`。
7. 普通用户和管理员密码统一使用 BCrypt 哈希保存，哈希强度使用通用配置 `PASSWORD_BCRYPT_STRENGTH`。
8. 私密文章第一阶段实现，访问规则包括公开、仅自己、仅好友、仅选中好友、排除选中好友。
9. 前台与后台共用一个 Vue 3 + Vite 前端应用，通过路由和布局分区。
10. Docker 方案后置为独立阶段。中间件容器、后端容器、前端容器都归属于本项目部署方案，不依赖当前公共服务容器。
11. Git 工作流按约定使用 `codex/CreatorSpace` 分支。
12. AI 功能暂不进入第一阶段功能范围，但 `.env` 先预留智谱 AI 配置，默认多模态模型为 `glm-4.6v-flash`。
13. 如果项目需要加入 AI 智能体，业务定位放在后台 CMS 的“创作与运营助手”中，优先服务文章、作品、标签、评论审核和运营分析，不在第一阶段做成独立前台聊天机器人。

## 3. 产品目标

### 3.1 核心目标

1. 建立一个可长期维护的个人内容平台。
2. 支持文章、作品、灵感、页面配置和主题风格统一管理。
3. 提供漂亮、可切换风格的访客前台。
4. 提供完整后台 CMS，让内容创建、管理、统计和配置都可视化完成。
5. 预留高级能力扩展空间，例如 Redis 统计、文章版本、搜索系统、AI 辅助写作和部署自动化。

### 3.2 终极产品形态

CreatorSpace 的终极目标不是做一个普通文章列表，也不是一个只放项目截图的简历页，而是做成“个人主题空间”。访客进入后应该感觉是在浏览一个有作者气质的长期创作宇宙: 文章像主题档案，作品像策展展品，灵感像写作桌上的碎片，后台则像一个能持续维护这个空间的内容工作台。

参考优秀个人作品集和创意站点的共性后，CreatorSpace 最终需要形成以下产品气质:

1. 首页是主题入口，不是营销页。第一屏需要明确展示站点主人、主题气质、文章入口、作品入口和登录/注册入口。
2. 文章不是时间倒序堆叠，而是按主题、系列、标签、阅读路径和封面风格组织。
3. 作品不是项目清单，而是带封面、背景、过程、截图、外链、时间线和创作说明的作品档案。
4. 灵感墙不是代码片段仓库，而是用于沉淀摘句、参考图、封面草图、选题、链接、Prompt 和未成文想法的素材墙。
5. 主题系统不是简单换色皮肤，而是覆盖布局密度、字体气质、封面规则、动效强度、卡片形态和首页模块排序的系统能力。
6. 后台 CMS 不是临时表单集合，而是内容生产工作台: 写作、归档、标签、媒体、评论、统计和主题配置都能在后台完成。
7. 数据结构应尽量一次性按最终实体关系设计，后续优先通过兼容字段、扩展表、配置 JSON 和新增索引演进，避免频繁破坏既有迁移。

设计参考来源:

- Awwwards Portfolio: https://www.awwwards.com/websites/portfolio/
- Figma Portfolio Website Examples: https://www.figma.com/resource-library/portfolio-website-examples/
- Creative Bloq Portfolio Examples: https://www.creativebloq.com/portfolios/examples-712368
- Webflow Portfolio Examples: https://webflow.com/blog/web-design-portfolios

### 3.3 终极功能边界

终极版本需要包含以下稳定能力边界:

| 模块 | 终极能力 |
| --- | --- |
| 站点与主页 | 站点资料、导航、社交链接、首页模块编排、SEO 配置、页脚信息、公告和联系方式 |
| 主题系统 | 主题预设、主题变量、字体/颜色/卡片/动效配置、主题预览、主题版本 |
| 文章系统 | 草稿、发布、私密可见性、分类、标签、系列、阅读路径、封面、目录、版本、推荐、置顶 |
| 作品系统 | 作品档案、封面、截图、技术栈、Demo/GitHub/视频链接、过程记录、时间线、推荐 |
| 灵感墙 | 摘句、图片、链接、Prompt、草图、参考资料、可见性、来源、标签和主题关联 |
| 媒体资源 | 本地文件资源、引用关系、封面图、作品图、附件、文件元信息和可见性 |
| 互动系统 | 评论、回复、点赞、收藏、留言板、评论审核、敏感词过滤 |
| 搜索系统 | 跨文章、作品、灵感、标签、分类的站内搜索，优先 PostgreSQL 原生能力 |
| 统计系统 | 访问日志、阅读量、点赞量、收藏量、热门内容、后台概览 |
| 用户系统 | 游客、注册用户、好友关系、管理员、编辑者预留、JWT 登录态 |
| AI 助手 | 后台写作建议、标签建议、摘要建议、评论审核建议、运营建议，默认后置实现 |
| 部署运维 | Docker 启动、健康检查、数据卷、迁移、日志、备份和 Windows PowerShell 文档 |

### 3.4 非目标

初版不追求一次性完成所有高级功能。以下内容可以放到后续阶段:

- AI 智能体、AI 写作、AI 摘要、AI 标签生成。
- 全文搜索引擎独立部署，例如 MeiliSearch 或 Elasticsearch。
- WebSocket 实时通知。
- 多租户站点。
- 复杂会员体系或商业支付。

## 4. 用户角色

| 角色 | 说明 | 核心权限 |
| --- | --- | --- |
| 游客 | 未登录访客 | 浏览公开内容、搜索 |
| 注册用户 | 已注册访客 | 浏览公开内容、评论、回复、点赞、收藏 |
| 管理员 | 站点拥有者 | 管理所有内容、配置、主题、文件、评论和统计 |
| 编辑者 | 可选角色 | 管理文章、作品和创意卡片 |
| 友人 | 可选角色 | 查看作者按好友关系开放的私密内容 |

第一版实现游客、注册用户、好友关系和管理员。编辑者作为权限模型预留。

## 5. 前台访客端需求

### 5.1 首页

首页需要体现个人品牌和内容入口，建议包含:

- 头像、昵称、一句话介绍。
- 精选文章。
- 精选作品。
- 最近动态或创作时间线。
- 技术栈标签。
- 标签云。
- 访问统计摘要。
- 社交链接或联系方式。

首页内容应由后台配置，不建议硬编码在前端。

### 5.2 博客文章页

文章页需要支持:

- 文章列表。
- 分类筛选。
- 标签筛选。
- 关键词搜索入口。
- 推荐 / 置顶文章。
- 草稿不对外展示。
- 私密文章需要访问控制，第一阶段实现公开、仅自己、仅好友、仅选中好友、排除选中好友五种模式。

### 5.3 文章详情页

文章详情页需要支持:

- Markdown 渲染。
- 代码高亮。
- 文章目录。
- 封面图。
- 摘要。
- 分类与标签展示。
- 阅读量统计。
- 点赞。
- 评论列表。
- 上一篇 / 下一篇。

### 5.4 作品集页

作品集页用于展示个人项目、设计、摄影、视频或实验作品，需要支持:

- 作品列表。
- 类型筛选。
- 技术栈筛选。
- 推荐作品。
- 作品封面。
- GitHub 链接。
- Demo 链接。
- 视频链接。
- 展示 / 隐藏状态。

### 5.5 作品详情页

作品详情页需要支持:

- 项目简介。
- 技术栈标签。
- 项目截图。
- 开发过程。
- 遇到的问题与解决方案。
- 项目时间线。
- 外部链接。

### 5.6 创意墙

创意墙用于展示碎片灵感、Prompt、图片、UI 截图、短代码、链接或项目想法。页面形态建议接近瀑布流或 Gallery。

需要支持:

- 卡片列表。
- 卡片类型筛选。
- 标签筛选。
- 图片 / 文本 / Prompt / 代码 / 链接类型。
- 来源链接。
- 是否公开。

### 5.7 关于页

关于页需要支持:

- 个人介绍。
- 技能栈。
- 经历。
- 联系方式。
- 社交链接。
- 可选简历下载入口。

内容应优先来自后台页面配置。

### 5.8 留言板 / 评论互动

需要支持:

- 文章评论。
- 作品评论。
- 多级回复评论。
- 评论审核后展示。
- 评论点赞。

游客只有浏览权限，评论、回复、点赞和收藏都要求登录。第一阶段评论系统直接按完整多级评论设计，并接入敏感词表。

### 5.9 搜索页

搜索范围建议包括:

- 文章标题。
- 文章摘要。
- 文章内容。
- 作品标题。
- 作品描述。
- 创意卡片标题。
- 标签。
- 分类。

第一版使用 PostgreSQL `pg_trgm` 扩展配合 GIN 索引实现模糊搜索，避免普通 `%keyword%` 查询在数据量增长后频繁全表扫描。后续如果需要更强的分词、排序和跨字段相关性，再评估 PostgreSQL 全文搜索、MeiliSearch 或 Elasticsearch。

## 6. 后台 CMS 需求

### 6.1 登录与注册

后台需要独立登录入口。前台需要注册和登录能力，后台需要管理员登录能力。

需要支持:

- 用户名 / 密码登录。
- 用户注册，注册字段只包含用户名和密码。
- JWT 登录态。
- BCrypt 密码哈希。
- 普通用户和管理员共用 `PASSWORD_BCRYPT_STRENGTH` 控制哈希强度。
- 用户不需要邮箱字段，用户名允许后续修改。
- 内置管理员账号 `admin`。
- 管理员初始密码哈希从本地 `.env` 读取，源密码不写入仓库。
- 登录状态过期处理。
- 未登录时禁止访问后台接口。

### 6.2 仪表盘

后台首页需要提供运营概览:

- 文章总数。
- 作品总数。
- 评论总数。
- 今日访问量。
- 最近 7 天访问趋势。
- 热门文章 Top 5。
- 热门作品 Top 5。
- 最新留言 / 评论。
- 标签分布。

### 6.3 文章管理

需要支持:

- 新建文章。
- 编辑文章。
- 删除文章。
- 草稿保存。
- 发布文章。
- 取消发布。
- 定时发布, 后续阶段实现。
- 置顶文章。
- 推荐文章。
- 私密文章, 第一阶段实现公开、仅自己、仅好友、仅选中好友、排除选中好友。
- Markdown 编辑。
- 封面上传。
- 分类与标签绑定。
- 文章版本保存与回滚, 后续阶段实现。

### 6.4 分类与标签管理

分类需要支持:

- 新增、编辑、删除。
- 排序。
- 启用 / 禁用。
- 用于文章或作品的类型区分。

标签需要支持:

- 新增、编辑、删除。
- 标签颜色。
- 标签权重或排序。
- 文章、作品、创意卡片复用。

### 6.5 作品管理

需要支持:

- 新增作品。
- 编辑作品。
- 删除作品。
- 设置展示 / 隐藏。
- 设置推荐。
- 上传封面。
- 管理项目截图。
- 配置 GitHub、Demo、视频等链接。
- 维护技术栈。
- 维护项目详情 Markdown。

### 6.6 创意墙管理

需要支持:

- 新增创意卡片。
- 编辑创意卡片。
- 删除创意卡片。
- 设置公开 / 私密。
- 上传图片。
- 维护来源链接。
- 选择卡片类型。
- 绑定标签。

### 6.7 评论管理

需要支持:

- 查看评论列表。
- 按目标类型筛选。
- 审核通过。
- 驳回。
- 删除。
- 拉黑用户名 / IP, 后续阶段实现。
- 敏感词过滤。

### 6.8 主题管理

主题系统是本项目特色能力。后台需要支持:

- 查看主题列表。
- 切换当前主题。
- 编辑主题基础信息。
- 配置主色、背景、字体、卡片样式和布局类型。
- 保存扩展 JSON 配置。
- 预览主题效果, 后续阶段实现。

预设主题建议:

| 主题 | 风格 |
| --- | --- |
| Cyber Night | 暗色科技风 |
| Soft Blog | 温柔浅色风 |
| Pixel World | 像素复古风 |
| Minimal White | 极简白色风 |
| Glass Space | 玻璃星空风 |

### 6.9 页面配置

页面配置用于避免把个人信息硬编码在前端。

需要支持:

- 站点名称。
- 站点副标题。
- 头像。
- 首页介绍文案。
- 社交链接。
- SEO 标题。
- SEO 描述。
- 首页精选内容配置。
- 关于页内容。

### 6.10 文件资源管理

需要支持:

- 上传文件。
- 上传文章封面。
- 上传作品封面。
- 上传作品截图。
- 文件列表。
- 文件预览。
- 文件分类。
- 文件大小限制。
- 文件引用统计。
- 删除文章或作品时，自动删除未被任何内容引用的文件。
- 同一图片被多个内容使用时按不同文件资源保存，不做跨内容复用。
- 本机本地存储, 路径相对项目根目录, 第一阶段不接入对象存储。

### 6.11 访问统计

需要支持:

- 记录访问路径。
- 记录访问目标类型。
- 记录 IP。
- 记录 User-Agent。
- 记录 Referer。
- 统计 PV。
- 统计 UV。
- 热门文章。
- 热门作品。
- 最近 7 天 / 30 天趋势。

第一版可以先写入 PostgreSQL。后续使用 Redis 做计数缓冲，再定时同步到 PostgreSQL。

### 6.12 操作日志

后台关键操作需要留痕:

- 新增文章。
- 更新文章。
- 删除文章。
- 发布文章。
- 切换主题。
- 审核评论。
- 上传文件。

后续可以通过 AOP 注解统一记录操作日志。

### 6.13 AI 创作与运营助手

如果需要在项目中加入 AI 智能体，建议不要把它单独做成一个和博客业务脱节的聊天页，而是放在后台 CMS 内，定位为“创作与运营助手”。它属于内容管理业务的增强能力，主要帮助管理员更快完成内容生产、作品包装、评论审核和运营分析。

业务归属:

- 主归属: 后台 CMS。
- 具体入口: 后台侧边栏新增“AI 助手”或在文章、作品、评论、仪表盘页面提供“AI 辅助”按钮。
- 服务对象: 管理员和可选编辑者。
- 不建议第一阶段开放给游客或普通用户。

建议能力:

- 文章助手: 根据标题或草稿生成摘要、优化标题、生成标签、检查错别字和语气。
- 作品助手: 根据项目描述、截图或技术栈生成作品介绍、亮点总结和展示文案。
- 标签助手: 根据文章、作品和创意卡片内容推荐标签和分类。
- 评论审核助手: 对评论做敏感内容提示、疑似垃圾评论提示和审核建议，最终决定仍由管理员完成。
- 运营助手: 根据访问统计、热门文章、热门作品给出内容更新建议。
- 搜索问答助手: 后续可以在前台提供基于站内公开内容的问答，但必须只回答公开内容，不能泄露私密文章或后台数据。

第一版 AI 智能体不需要复杂工作流。建议先实现为“单轮任务型助手”:

1. 管理员在后台选择一个内容对象，例如文章草稿或作品。
2. 选择 AI 动作，例如生成摘要、生成标签、优化标题或生成作品介绍。
3. 后端调用智谱 AI。
4. 前端展示 AI 建议。
5. 管理员手动采纳、编辑或丢弃。

后续如果要体现“智能体”概念，可以再扩展任务记录、上下文记忆、工具调用和多步计划。但课堂展示阶段重点是说清楚它服务内容系统，而不是为了 AI 单独增加复杂系统。

### 9.2 最终数据库实体边界

为了支撑终极产品形态，数据库最终应覆盖以下实体边界:

| 领域 | 建议表 | 说明 |
| --- | --- | --- |
| 用户与权限 | `users`、`roles`、`user_roles`、`user_profiles`、`user_friendships` | `users` 保存登录身份，`user_profiles` 保存展示资料，好友关系服务私密内容 |
| 站点配置 | `site_configs`、`site_profiles`、`navigation_items`、`social_links`、`page_configs` | 将首页、导航、页脚、社交链接和页面 SEO 从前端硬编码中拆出 |
| 主题系统 | `theme_configs`、`theme_variants`、`theme_assets`、`theme_versions` | 主题变量建议使用 `jsonb` 保存颜色、字体、布局、动效和卡片规则 |
| 内容分类 | `categories`、`tags`、`tag_aliases` | 分类按模块隔离，标签允许权重、颜色、别名和展示排序 |
| 文章系统 | `articles`、`article_versions`、`article_tags`、`article_visibility_users`、`article_series`、`article_series_items`、`article_relations` | 系列和相关文章要成为正式能力，不只靠正文手写链接 |
| 作品系统 | `portfolio_projects`、`project_tags`、`project_images`、`project_milestones`、`project_links`、`project_process_notes` | 作品详情需要支撑截图、外链、过程、时间线和多类型媒体 |
| 灵感墙 | `inspiration_cards`、`inspiration_tags`、`inspiration_sources`、`inspiration_relations` | 灵感卡片可关联文章、作品或其他灵感，服务后续创作归档 |
| 内容块 | `content_blocks`、`content_block_assets` | 首页、关于页、专题页可用块式内容表达，减少硬编码页面 |
| 文件资源 | `file_resources`、`file_resource_references` | 统一管理封面、截图、附件、头像和站点素材 |
| 互动系统 | `comments`、`comment_reactions`、`like_records`、`favorite_records`、`guestbook_entries`、`sensitive_words` | 评论和留言都进入审核流程，点赞收藏按资源类型统一建模 |
| 搜索统计 | `visit_logs`、`content_statistics`、`search_logs`、`daily_metrics` | 原始访问日志和聚合指标分开，便于后台看板和性能优化 |
| 后台审计 | `operation_logs`、`admin_audit_snapshots` | 后台关键操作必须可追踪，重要配置变更可保存快照 |
| AI 预留 | `ai_agent_tasks`、`ai_agent_messages`、`ai_suggestions` | AI 输出作为建议保存，采纳动作进入操作日志 |

### 9.3 数据库设计原则

1. 所有核心业务表使用统一基础字段: `id`、`created_at`、`updated_at`、必要时包含 `deleted_at`、`created_by`、`updated_by`。
2. 文章、作品、灵感、评论、文件等可被访问的资源需要统一保留可见性、状态、排序和统计字段。
3. 主题、页面配置、首页模块、内容块等高度可变结构优先使用 `jsonb` 保存配置，但核心查询条件必须拆成普通列。
4. 标签、分类、系列、推荐、置顶、可见性名单等关系能力使用独立表，不把多个 ID 拼接到字符串字段。
5. 外链字段需要保存原始 URL、展示标题、链接类型和安全校验状态，前后端都只允许 `http` / `https` 外链。
6. 搜索字段需要提前考虑索引: 标题、摘要、slug、标签名、分类名、作品描述和灵感内容都应有可搜索路径。
7. 统计字段可以保留冗余计数，但真实访问日志和聚合统计要分表，避免列表页频繁扫描日志。
8. V1 迁移一旦进入共享环境，不再回改已执行结构；需要调整时新增 `V版本号__描述.sql`。

## 11. 分阶段范围

### 11.0 模块分工原则

后续开发按模块分工，不按零散页面分工。每个模块完成时应包含后端模型、接口、前端页面、基础数据、必要检查和文档更新。

| 模块 | 主要负责人方向 | 交付物 |
| --- | --- | --- |
| 数据库与迁移 | 后端 / 数据库 | V1 最终主实体、V2 展示数据、字段注释、索引、迁移验证 |
| 用户与权限 | 后端 | 注册、登录、JWT、角色、好友关系、私密内容访问控制 |
| 内容中台 | 后端 + 前端后台 | 文章、分类、标签、作品、灵感墙 CRUD 和发布流程 |
| 前台主题体验 | 前端 | 首页、文章、作品、灵感墙、关于页、主题切换和响应式体验 |
| 媒体资源 | 后端 + 前端后台 | 文件上传、资源管理、引用关系、封面和作品截图 |
| 互动与统计 | 后端 + 前端后台 | 评论、点赞、收藏、留言、访问日志、内容统计和后台看板 |
| 部署运维 | 全栈 | Docker Compose、镜像、健康检查、日志、数据卷、README 启动文档 |
| AI 增强 | 后端优先 | 后台建议型助手、AI 任务记录、采纳审计，默认后置 |

本项目后续不把“派遣子智能体 reviewer”作为必走流程。质量控制改为: 开发者自查、必要的本地类型检查/构建/后端测试、用户人工验收和 PR 差异审阅。

目标: 前后端项目能完整跑起来，并具备核心内容发布能力。

建议范围:

- MyBatis-Plus 接入。
- MyBatis-Plus 分页插件接入。
- JWT 鉴权。
- 用户注册。
- 内置管理员初始化。
- 管理员登录。
- 文章发布。
- 文章列表。
- 文章详情。
- 私密文章。
- 分类与标签。
- 作品展示。
- 文件上传基础能力。

### 11.4 第四阶段: AI / 全栈加分项

目标: 让平台具备更强的创作辅助能力。

建议范围:

- 后台 AI 创作与运营助手。
- 文章草稿摘要生成。
- AI 生成摘要。
- AI 生成标签。
- AI 优化标题。

- WebSocket 通知。

AI 智能体的业务边界:

- 优先放在后台 CMS，作为内容创作、作品包装、评论审核和运营分析助手。

- AI 输出只能作为建议，不自动发布文章、不自动删除评论、不自动修改主题配置。
- 所有 AI 采纳动作需要记录到后台操作日志。

### 11.5 部署阶段: Docker 独立方案

目标: 为 CreatorSpace 设计独立部署方案。

原则:

- 当前公共服务容器只作为开发环境参考，不作为项目最终部署依赖。
- PostgreSQL、Redis、后端、前端都归属于 CreatorSpace 自己的 Docker 方案。
- Docker 配置在中后期单独设计和验证，不阻塞第一阶段功能开发。
- 部署文档需要明确 Windows PowerShell 下的启动、停止、查看日志和数据保留方式。

## 12. 非功能需求

### 12.1 安全

- 后台接口必须鉴权。
- 管理员密码必须加密存储。
- 普通用户和管理员密码必须使用同一套 BCrypt 强度配置。
- JWT 密钥必须从 `.env` 或部署密钥配置读取。
- `.env` 必须加入 `.gitignore`，不得提交真实密钥。
- 智谱 API Key 等第三方密钥不得提交到仓库。
- 文件上传必须校验大小和类型。
- 评论内容需要防 XSS。
- Markdown 渲染需要处理 HTML 注入风险。
- 不在 README、日志、提交信息中暴露密钥。

### 12.2 性能

- 首页和文章列表需要分页。
- 列表分页统一使用 MyBatis-Plus 分页插件。
- 站内搜索优先使用 PostgreSQL `pg_trgm` + GIN 索引。
- 热门内容统计后续使用 Redis 降低数据库压力。
- 文件资源需要合理缓存。
- 后台数据看板需要避免每次全表扫描。

### 12.3 可维护性

- 前后端类型定义清晰。
- 后端统一响应结构。
- 后端统一异常处理。
- 后端模块边界清晰。
- 前端 API 调用集中管理。
- 文档中的命令必须适配 Windows PowerShell。

### 12.4 可观测性

- 后端需要基础日志。
- 关键后台操作需要操作日志。
- 后续可加入健康检查接口。
- 后续可加入接口耗时统计。

## 15. 环境变量规划

本项目使用 `.env` 保存本机开发配置和敏感配置。真实 `.env` 不提交仓库，提交 `.env.example` 作为模板。

已预留的配置分类:

- 应用运行: `APP_ENV`、`SPRING_PROFILES_ACTIVE`、`SERVER_PORT`。
- 前后端地址: `WEB_BASE_URL`、`API_BASE_URL`、`CORS_ALLOWED_ORIGINS`。
- 数据库: `POSTGRES_HOST`、`POSTGRES_PORT`、`POSTGRES_DB`、`POSTGRES_USERNAME`、`POSTGRES_PASSWORD`。
- Redis: `REDIS_HOST`、`REDIS_PORT`、`REDIS_DATABASE`、`REDIS_PASSWORD`。
- JWT: `JWT_SECRET`、`JWT_ACCESS_TOKEN_EXPIRE_MINUTES`、`JWT_REFRESH_TOKEN_EXPIRE_DAYS`。
- 密码哈希: `PASSWORD_BCRYPT_STRENGTH`。
- 管理员初始化: `ADMIN_USERNAME`、`ADMIN_PASSWORD_HASH`。
- 文件存储: `FILE_STORAGE_TYPE`、`LOCAL_FILE_STORAGE_ROOT`、`LOCAL_FILE_PUBLIC_PREFIX`、`UPLOAD_MAX_FILE_SIZE_MB`、`UPLOAD_MAX_REQUEST_SIZE_MB`。
- AI 预留: `AI_ENABLED`、`AI_PROVIDER`、`ZHIPU_API_BASE_URL`、`ZHIPU_API_KEY`、`ZHIPU_MULTIMODAL_MODEL`、`ZHIPU_TEXT_MODEL`、`ZHIPU_REQUEST_TIMEOUT_SECONDS`、`ZHIPU_MAX_OUTPUT_TOKENS`、`ZHIPU_THINKING_ENABLED`。
