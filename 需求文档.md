# CreatorSpace 终极目标需求

本文档只保留 CreatorSpace 的终极目标需求，不再记录阶段计划、当前环境快照、临时技术决策、接口草案或数据库草案。所有实现细节以代码、README 和专项技术文档为准。

## 1. 产品定位

CreatorSpace 是一个“个人主题风格博客与创意作品展示平台”。它不是普通文章列表，也不是只放项目截图的简历页，而是一个长期维护的个人主题空间:

- 文章像主题档案，用来沉淀长期思考、教程、随笔和创作记录。
- 作品像展厅展品，用来展示个人项目、实验、设计、影像或其他创意成果。
- 灵感像写作桌上的碎片，用来收藏摘句、链接、Prompt、参考图、选题和未成形想法。
- 创作中心像个人工作台，用来写博客、上传作品、管理文件和查看审核状态。
- 后台像审核与运营工作台，用来持续管理审核、主题、评论、文件、统计和站点配置。

最终体验要让访客感到自己进入了一个有作者气质、有主题审美、有作品陈列感的个人创作宇宙。

## 2. 终极产品形态

1. 首页是主题入口，不是营销页。首屏需要清楚表达站点主人、主题气质、文章入口、作品入口和登录/注册入口。
2. 文章不是时间倒序堆叠，而是按主题、系列、标签、阅读路径、封面风格和推荐关系组织。
3. 作品不是项目清单，而是带封面、截图、背景、过程、时间线、外链和创作说明的作品档案。
4. 灵感墙不是代码片段仓库，而是可检索、可归档、可关联文章和作品的创意素材墙。
5. 主题系统不是简单换色皮肤，而是覆盖布局密度、字体气质、封面规则、动效强度、卡片形态和首页模块排序的系统能力。
6. 创作中心不是只给管理员使用的后台入口，普通注册用户应能生产自己的博客和作品，并通过审核进入公开展示。
7. 后台 CMS 不是临时表单集合，而是管理员审核与运营工作台。审核、归档、推荐、标签、媒体、评论、统计和主题配置都应能在后台完成。
8. 数据结构应按最终实体关系设计，后续优先通过兼容字段、扩展表、配置 JSON 和新增索引演进，避免频繁破坏既有迁移。

## 3. 目标用户与角色

| 角色 | 说明 | 终极权限 |
| --- | --- | --- |
| 游客 | 未登录访客 | 浏览公开内容、搜索、查看公开作品和灵感 |
| 注册用户 | 已登录创作者与读者 | 发表和管理自己的博客/作品、上传创作图片资源、提交审核、评论、回复、点赞、收藏、留言、访问授权内容 |
| 友人 | 作者维护的关系用户 | 查看按好友关系开放的私密内容 |
| 编辑者 | 可选内容协作者 | 管理授权范围内的文章、作品和灵感 |
| 管理员 | 站点拥有者 | 审核用户提交内容、管理全站内容和用户、运营推荐、主题、评论、文件、统计和站点配置 |

## 4. 终极功能边界

| 模块 | 终极能力 |
| --- | --- |
| 站点与主页 | 站点资料、导航、社交链接、首页模块编排、SEO 配置、页脚信息、公告和联系方式 |
| 主题系统 | 主题预设、主题变量、字体/颜色/卡片/动效配置、主题预览、主题版本和主题切换 |
| 文章系统 | 普通用户草稿、提交审核、发布后公开展示、私密可见性、分类、标签、系列、阅读路径、封面、目录、版本、推荐和置顶 |
| 作品系统 | 普通用户作品草稿、提交审核、作品档案、封面、截图、技术栈、Demo/GitHub/视频链接、过程记录、时间线和推荐 |
| 灵感墙 | 摘句、图片、链接、Prompt、草图、参考资料、可见性、来源、标签和主题关联 |
| 媒体资源 | 普通用户图片资产、管理员附件资源、本地文件资源、引用关系、封面图、作品图、附件、文件元信息、预览和可见性 |
| 互动系统 | 评论、回复、点赞、收藏、留言板、评论审核、敏感词过滤和基础反垃圾策略 |
| 搜索系统 | 跨文章、作品、灵感、标签、分类和页面配置的站内搜索 |
| 统计系统 | 访问日志、阅读量、点赞量、收藏量、热门内容、趋势数据和后台概览 |
| 用户系统 | 游客、注册用户、好友关系、管理员、编辑者预留和登录态管理 |
| AI 助手 | 后台写作建议、标签建议、摘要建议、评论审核建议和运营建议 |
| 部署运维 | Docker 启动、健康检查、数据卷、迁移、日志、备份、回滚和 Windows PowerShell 文档 |

## 5. 前台体验需求

### 5.1 首页

首页要体现个人品牌、主题审美和内容入口。终极形态应包含:

- 作者头像、昵称、一句话介绍和站点主题气质。
- 精选文章、精选作品、灵感入口和最近动态。
- 技术栈标签、标签云、访问统计摘要和社交链接。
- 可由后台配置的首页模块排序、展示内容和主题样式。
- 适配桌面与移动端，并保留必要的动效降级能力。

### 5.2 文章体验

文章体验应支持:

- 文章列表、分类筛选、标签筛选和关键词搜索。
- 推荐文章、置顶文章、系列文章和阅读路径。
- Markdown 渲染、代码高亮、目录、封面、摘要和上一篇/下一篇。
- 阅读量、点赞、收藏和评论互动。
- 私密文章访问控制，包括公开、仅自己、仅好友、仅选中好友、排除选中好友。

### 5.3 作品展厅

作品展示应支持:

- 作品列表、类型筛选、技术栈筛选和推荐作品。
- 作品封面、截图、项目简介、技术栈、过程记录和时间线。
- GitHub、Demo、视频、文章复盘等外部链接。
- 作品详情页需要像“展品档案”，而不是普通 CRUD 详情。

### 5.4 灵感墙

灵感墙应支持:

- 瀑布流或 Gallery 式卡片展示。
- 文本、图片、Prompt、代码、链接、草图和参考资料类型。
- 标签、来源、可见性和与文章/作品的关联。
- 后续能从灵感沉淀为文章或作品过程记录。

### 5.5 关于页与搜索页

关于页应由后台配置，展示个人介绍、技能、经历、联系方式、社交链接和可选简历入口。

搜索页应覆盖文章、作品、灵感、标签、分类和可公开页面配置，并提供清晰的内容类型区分。

## 6. 后台 CMS 需求

后台 CMS 的终极目标是让管理员可以在一个稳定、清晰、高效的工作台里审核内容、维护运营位并管理整个个人主题空间。普通用户的写作、作品上传和资源管理应主要发生在前台创作中心。

### 6.1 内容审核与运营

- 审核通过、驳回、发布、撤回用户提交的文章。
- 管理封面、分类、标签、系列、私密可见性和文章版本。
- 审核通过、驳回、隐藏、推荐用户提交的作品。
- 管理作品截图、技术栈、外链、过程记录和时间线。
- 新增、编辑、删除灵感卡片，并维护类型、标签、来源和可见性。

### 6.2 内容组织

- 管理分类、标签、标签颜色、排序、权重和别名。
- 管理文章系列、阅读路径和关联内容。
- 管理首页精选内容、推荐内容和置顶内容。
- 管理页面配置，避免个人信息硬编码在前端。

### 6.3 互动与审核

- 查看评论、留言和多级回复。
- 审核通过、驳回、删除和标记敏感评论。
- 支持敏感词过滤、基础反垃圾策略和后续黑名单能力。
- 评论、点赞、收藏和留言都应能被统计和追踪。

### 6.4 文件与媒体

- 上传文章封面、作品封面、作品截图、头像和附件。
- 查看文件列表、预览文件、管理文件分类和文件大小限制。
- 维护文件引用关系，避免误删仍被内容使用的资源。
- 文件存储以本地存储为基础，并保留对象存储扩展空间。

### 6.5 主题与站点配置

- 查看主题列表、切换当前主题、编辑主题基础信息。
- 配置主色、背景、字体、卡片样式、布局类型和动效强度。
- 保存扩展 JSON 配置，支持主题预览和主题版本。
- 管理站点名称、副标题、头像、首页文案、SEO、导航、社交链接和页脚信息。

### 6.6 数据看板与审计

- 展示文章数、作品数、评论数、访问量、热门文章、热门作品和趋势数据。
- 记录后台关键操作，例如新增、更新、删除、发布、切换主题、审核评论和上传文件。
- 后续可加入接口耗时、错误率、访问来源和内容转化分析。

## 7. AI 助手终极边界

AI 能力应服务内容生产和运营，不应脱离博客业务单独变成聊天玩具。

终极能力包括:

- 根据文章草稿生成摘要、标题建议、标签建议和语气优化。
- 根据作品描述、截图或技术栈生成作品介绍和亮点总结。
- 根据评论内容给出敏感风险提示和审核建议。
- 根据访问统计、热门内容和搜索日志给出运营建议。
- 后续可提供前台公开内容问答，但必须遵守内容可见性，不能泄露私密文章或后台数据。

AI 输出只能作为建议，不自动发布文章、不自动删除评论、不自动修改主题配置。所有采纳动作应记录到操作日志。

## 8. 非功能需求

### 8.1 安全

- 后台接口必须鉴权。
- 密码必须使用 BCrypt 等安全哈希保存。
- JWT 密钥、数据库密码、Redis 密码和第三方 API Key 只能放在本地 `.env` 或部署密钥系统。
- 不在仓库、README、日志、提交信息或 PR 描述中暴露真实密钥。
- 文件上传必须校验大小、类型和访问路径。
- Markdown 与评论内容必须防 XSS。
- 私密内容访问必须由后端校验，不依赖前端隐藏。

### 8.2 性能

- 首页、列表页和后台列表必须分页或限制返回量。
- 搜索需要有索引或可扩展的搜索方案，避免数据增长后全表扫描。
- 热门内容、访问统计和计数类能力可使用 Redis 或聚合表降低数据库压力。
- 静态资源、上传文件和前端构建产物需要合理缓存。

### 8.3 可维护性

- 前后端模块边界清晰，接口响应结构统一。
- 配置集中管理，个人信息和站点配置尽量从后台或 `.env` 读取。
- 数据库迁移一旦进入共享环境，不原地修改已执行脚本。
- 文档中的命令必须适配 Windows PowerShell。
- Docker 部署必须说明启动、停止、日志、数据卷、备份和回滚方式。

## 9. 终极验收标准

1. 访客可以完整浏览首页、文章、作品、灵感墙、搜索页和关于页。
2. 注册用户可以登录、发表博客、上传创意作品、提交审核、管理自己的图片资源、评论、回复、点赞、收藏和留言。
3. 管理员可以在后台完成文章/作品审核、运营推荐、灵感、分类、标签、评论、文件、主题和站点配置管理。
4. 私密文章和授权内容只能被符合权限的用户访问。
5. 首页内容、关于页内容、导航、社交链接和主题配置不再依赖前端硬编码。
6. 作品详情具备展厅式表达，包括封面、截图、技术栈、外链、时间线和创作说明。
7. 灵感墙能沉淀碎片素材，并能与文章或作品产生关联。
8. 后台有真实统计、热门内容、访问趋势和操作日志。
9. Docker 部署可通过根目录 `.env` 启动，并保留数据库、Redis 和上传文件数据。
10. README 和部署文档能让新维护者按 Windows PowerShell 命令完成本地启动、Docker 启动、检查和回滚。

## 10. 非目标

以下内容不是终极目标的核心，不作为当前产品判断标准:

- 多租户建站系统。
- 商业支付、会员订阅或复杂电商能力。
- 面向公众开放的通用 AI 聊天平台。
- 与个人创作展示无关的企业级协同办公功能。
- 为追求技术炫技而脱离内容平台定位的复杂架构。
